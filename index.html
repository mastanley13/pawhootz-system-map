Journey Lanes

The journey view keeps the architecture tied to real service operations

System decisions only matter if they improve the real client and staff journeys. These lanes show where the current build already carries weight and where the operating model still needs closure.

New Household

Onboarding, pets, vet info, and vaccine readiness

Current: Clients can register, log in, add pets, manage household data, and upload vaccine documents.

Target: A closed-loop onboarding path with application completion, policy gates, vaccine readiness, and clear next-step messaging.

Closure: Tie new-client application proof, waiver status, and payment profile requirements into the same readiness model.

  1. Register household account
  2. Add pet and veterinarian information
  3. Upload vaccine records
  4. Complete required application or waivers
  5. Unlock service-specific booking gates

Boarding Client

Boarding selection, eligibility, and stay lifecycle

Current: Clients can move through boarding service selection and appointment creation, but the final stay-state model is still unresolved.

Target: A boarding flow with deposits, forms, lodging logic, pre-arrival readiness, in-stay visibility, and clean check-out follow-up.

Closure: Close the stay-model decision before tying webcams, report cards, and communications tightly to boarding state.

  1. Pass boarding eligibility and form gates
  2. Select lodging and add-ons
  3. Create the correct booking or stay record
  4. Expose staff-operating states during the stay
  5. Deliver departure follow-up and recap

Daycare Client

Evaluation-first daycare model

Current: Daycare currently behaves more like package selection plus staff follow-up than a finished self-serve flow.

Target: An intentional evaluation, approval, package, payment, and attendance model that matches how PawHootz actually operates.

Closure: Choose whether daycare remains request-first or becomes partially self-serve after evaluation approval.

  1. Express interest or select package
  2. Trigger evaluation and readiness checks
  3. Approve yard or attendance eligibility
  4. Collect payment and package state
  5. Allow the correct scheduling path

Grooming + Training Client

Service booking with reminders, payment, and follow-up

Current: Grooming and training booking flows already exist and push real operational payloads.

Target: Production service flows with final pricing logic, reminder templates, deposits, and post-service follow-up.

Closure: Export and prove the GHL-side reminder, deposit, and follow-up behaviors instead of assuming they are configured correctly.

  1. Choose service or package
  2. Validate readiness and eligibility
  3. Book the correct calendar slot
  4. Send confirmations and reminders
  5. Close the loop with notes, payment, and service recap

Data Model + System Boundaries

The final build needs explicit ownership by object, not just by page

This is where the map moves from UI inventory to system architecture. Each record type needs a clear home and a clear rule for who is allowed to mutate it.

Household

Contact remains the household anchor

Primary client identity, communication preferences, payment readiness, and household-level onboarding state should live here. Avoid burying operational pet state inside contact notes or ad-hoc fields.

Pet

Pets are the canonical pet profile

Breed, size, service attributes, behavioral notes, icons, and service eligibility should resolve through the pet object rather than duplicated field clusters.

Vaccine

Vaccine records stay document-centric

Each vaccine should preserve type, dates, document source, and review status so the approval and expiry loop remains auditable.

Appointment

Appointments should own the scheduled service instance

Use appointments for confirmed scheduling. Do not overload them with every operational fact if the stay or attendance model becomes more complex.

Report

Reports should own service recap content

Report cards, photos, and narrative recap content should have one explicit home, one creation workflow, and one client-delivery path.

Reference

Veterinarians and pet icons are already useful support objects

These live objects show that PawHootz is already beyond a single-object CRM design. Keep using that advantage intentionally.

Major Boundary Decision Still Open

  • Boarding and possibly daycare still need a clear decision on whether appointments are sufficient or whether a true stay or attendance record is needed.
  • That decision affects webcams, report cards, operational status, and launch reporting.

What Is Already Working In Our Favor

  • The live location already uses custom objects instead of flattening everything into contact fields.
  • That gives us a cleaner path to a durable operating system if we keep the boundaries disciplined.

Decision Register

These are the decisions worth closing before spending too much more effort on polish

The highest-risk architecture gaps are not visual. They are control decisions that affect every downstream build choice.

Critical

Replace the current client auth model

Recommendation: Move away from contact-field password handling and adopt a proper credential pattern with reset and security hygiene.

Why it matters: Without this change, the client portal remains functionally impressive but structurally unsafe for launch.

Critical

Close the appointments-versus-stay decision

Recommendation: Use appointments for scheduling, then add a durable stay or attendance state model if boarding and daycare operations need more than time slots.

Why it matters: This decision controls webcams, report cards, operational status, and reporting clarity.

Critical

Define the final daycare operating model

Recommendation: Keep daycare request-first until evaluation, approval, package state, and payment rules are explicitly documented.

Why it matters: If this stays fuzzy, the portal will keep oscillating between self-serve and staff-managed logic.

High

Prove deposit and payment enforcement

Recommendation: Capture direct GHL or checkout proof for deposit logic before treating any service flow as launch-ready.

Why it matters: Docs may say deposits exist, but the launch risk lives in what the system actually enforces.

High

Formalize status truth in staff operations

Recommendation: Reduce note-driven status handling and define which surface owns operational truth for appointments, stays, and completion states.

Why it matters: If status is not disciplined, downstream automation and reporting will remain brittle.

Cutover Sequence

The map should end in an executable sequence, not just an architecture ideal

A Champs-level artifact needs to show not only where the system is heading, but how it gets there without creating confusion during launch.

Phase 1

Freeze the operating model decisions

Close auth direction, daycare model, stay-state design, and payment or deposit posture so the build stops branching in multiple directions.

  • Approve final auth approach
  • Approve daycare operating model
  • Approve stay-state ownership
  • Confirm deposit rules by service line

Phase 2

Lock configuration truth and evidence

Capture GHL exports, screenshot proof, and source references so the map reflects what is actually configured.

  • Export workflows and templates
  • Capture form and calendar notification settings
  • Attach screenshot or export proof to each open evidence gap
  • Refresh the live GHL summary snapshot

Phase 3

Harden launch-critical surfaces

Focus engineering and configuration work on security, state discipline, and service flows that would break trust during launch.

  • Replace insecure auth handling
  • Close report-card and webcam gating loops
  • Finalize payment and deposit enforcement
  • Reduce note-driven operational truth where possible

Phase 4

Run cutover rehearsal and sign-off

Treat launch like an operating event, not just a deployment. Prove the runbook before committing to the final switchover.

  • Create dry-run migration checklist
  • Run staff acceptance walk-throughs
  • Capture rollback posture and owner approvals
  • Set launch freeze and go-live checkpoints
